# 需求文档: 全仓发布级深度优化

## 1. 概述

### 1.1 一句话描述

对中关村自主大模型产业联盟官网执行一次覆盖产品体验、品牌内容、前端工程、CI/CD、文档和测试的全仓审计与整改，并以可复核的质量证据将仓库收口到可合并、可部署、可正式对外发布的状态。

### 1.2 核心价值

- 将“页面能访问”提升为“联盟级官网可信、专业、清晰且具有行动号召力”。
- 消除影响用户决策、品牌可信度、可访问性、搜索收录、发布稳定性和后续维护效率的系统性问题。
- 把分散在既有设计稿、历史规格、代码、测试和部署配置中的质量要求统一为一套可持续执行的发布基线。
- 通过审计、整改、回归验证和发布证据闭环，避免只产出问题清单而不解决问题。

### 1.3 目标用户

- **主要用户**：希望了解联盟、参与工作组、申请加入、寻求合作或获取联盟动态的机构代表、专业用户、产业伙伴与公众访客。
- **次要用户**：媒体、研究人员、政策与产业观察者，以及中文或英文访问者。
- **内部用户**：负责内容维护、代码贡献、评审、部署和长期治理的联盟官网维护者。

### 1.4 治理原则

1. **用户任务优先**：围绕“理解联盟—建立信任—找到适合的参与方式—完成行动”优化，而非仅追求视觉变化。
2. **事实边界优先**：可以重写表达、调整层级与强化号召，但不得虚构成员、数据、荣誉、政策背书、合作关系或授权状态。
3. **统一而非堆叠**：优先统一信息架构、设计语言、内容口径与工程基线，不以新增更多组件、依赖或文档数量作为优化成果。
4. **证据驱动**：所有整改项应能通过自动化检查、人工走查、截图、报告或可重复命令验证。
5. **发布可逆**：保留有效公开 URL 和关键外部入口；必须变更时提供兼容路径或明确重定向方案。

### 2.2 验收标准

#### R-001: 全仓基线审计与问题闭环

- **WHEN** 进入实施阶段，**THEN** 项目 **SHALL** 建立覆盖全部静态路由、动态路由模板、中英文版本、主题、主要视口、交互状态、内容源、测试、文档和工作流的审计矩阵。
- **WHEN** 审计发现问题，**THEN** 项目 **SHALL** 以 P0/P1/P2、影响范围、复现证据、整改方式和验证结果记录并闭环；发布前不得遗留 P0，P1 必须修复或经用户明确接受。
- **WHEN** 既有规格、设计稿和当前实现发生冲突，**THEN** 项目 **SHALL** 基于本需求、事实来源和当前用户价值作出裁决，并更新或标记过时文档。

#### R-002: 信息架构与全站 UI 统一

- **WHEN** 用户首次进入任一主要落地页，**THEN** 页面 **SHALL** 在首屏明确说明页面主题、核心价值和下一步行动，不使用只有内部人员才能理解的抽象口号。
- **WHEN** 用户浏览页头、页尾、Hero、内容区块、卡片、按钮、标签、空态和详情页，**THEN** 系统 **SHALL** 使用统一的视觉层级、设计 token、排版、间距、状态反馈和组件语义。
- **WHEN** 页面结构或视觉语言调整，**THEN** 系统 **SHALL** 保持品牌识别、内容可读性和跨页面一致性，并移除无目的装饰、重复信息和误导性视觉暗示。
- **IF** 现有页面或区块不能支持用户任务，**THEN** 项目 **SHALL** 允许重组、合并或重设计，但必须维护有效 URL 的兼容性。

#### R-003: 全路径 UX、响应式与无障碍治理

- **WHEN** 用户在手机、平板、笔记本和宽屏设备上访问，**THEN** 页面 **SHALL** 无横向溢出、遮挡、不可触达控件、异常换行或信息层级坍塌。
- **WHEN** 用户通过键盘、触控、鼠标、减少动态效果偏好或屏幕阅读器操作，**THEN** 导航、主题/语言切换、展开面板、链接和 CTA **SHALL** 可理解、可操作、焦点可见且状态可感知。
- **WHEN** 执行自动与人工无障碍检查，**THEN** 站点 **SHALL** 以 WCAG 2.2 AA 为最低目标，并处理语义结构、名称/角色/值、对比度、焦点顺序、跳转链接、点击目标和动态效果问题。
- **WHEN** JavaScript 延迟或不可用且功能本可由静态 HTML 完成，**THEN** 核心内容和导航 **SHALL** 仍可访问。

#### R-004: 联盟级中文与英文内容升级

- **WHEN** 审阅公开文案，**THEN** 项目 **SHALL** 统一联盟定位、术语、语气、标题层级、行动动词和中英文专有名词，删除空泛、重复、机械翻译或夸大表达。
- **WHEN** 重写首页及关键转化页面，**THEN** 内容 **SHALL** 清楚回答“联盟是谁、聚焦什么、为何值得参与、谁适合参与、如何参与、提交后会怎样”。
- **WHEN** 内容涉及成员、牵头单位、专家、数字、成果、政策或合作关系，**THEN** 文案 **SHALL** 仅使用仓库中可追溯且允许公开的信息；不能确认时使用中性角色描述或明确待确认状态。
- **WHEN** 中文内容发生实质调整，**THEN** 对应英文页面 **SHALL** 保持信息等价、术语一致和自然表达；中文作为事实权威源，英文不得新增中文未声明的承诺。
- **WHEN** CTA 出现，**THEN** 文案 **SHALL** 准确描述动作与去向，避免多个不同目标共用含糊的“立即加入”等标签。

#### R-005: 转化路径与外部申请体验优化

- **WHEN** 用户选择加入联盟、参与工作组、专业用户计划或机构生态共建，**THEN** 页面 **SHALL** 明确适用对象、参与价值、必要条件、流程、外部跳转和隐私边界。

- **WHEN** 同一 CTA 在桌面导航、移动菜单、正文与页尾出现，**THEN** 其目标、名称和行为 **SHALL** 一致且可被自动化测试覆盖。

- **WHEN** 检查站内链接、资源和动态路由，**THEN** 系统 **SHALL** 不存在可复现的内部死链、孤立关键页面、错误语言跳转或静态导出遗漏。
- **WHEN** 对代表性页面执行实验室质量检查，**THEN** 移动端与桌面端 Lighthouse 目标 **SHALL** 达到 Performance ≥ 90、Accessibility ≥ 95、Best Practices ≥ 95、SEO ≥ 95；无法稳定达到时必须记录环境、瓶颈和经用户接受的例外。
- **WHEN** 运行浏览器兼容性回归，**THEN** 中文与英文关键路径 **SHALL** 在 Chromium、Firefox、WebKit 及代表性移动视口通过。
- **WHEN** 审查依赖、响应头能力和公开前端配置，**THEN** 项目 **SHALL** 不暴露密钥或个人信息，不引入已知高危/严重生产依赖漏洞，并在静态托管能力范围内采用合理安全基线。

#### R-007: 代码与依赖债务治理

- **WHEN** 审计源码，**THEN** 项目 **SHALL** 识别并处理重复组件/内容、死代码、无效样式、过度客户端化、脆弱条件分支、类型逃逸、命名不一致和不必要依赖。
- **WHEN** 重构页面与组件，**THEN** 公开内容 **SHALL** 继续集中在 `src/content/`，站点配置集中在 `src/config/site.ts`，默认使用 React Server Components，仅在真实浏览器交互需要时使用 Client Components。
- **WHEN** 升级、替换或删除依赖，**THEN** 项目 **SHALL** 说明必要性、验证锁文件与双构建目标，并避免为单一简单能力引入高成本依赖。
- **WHEN** 完成整改，**THEN** TypeScript 严格检查和 ESLint **SHALL** 无错误，不得以新增宽泛忽略、`any` 或降低规则掩盖债务。

#### R-008: CI/CD 发布闸门优化

- **WHEN** PR 或 `main` 触发流水线，**THEN** CI **SHALL** 使用 Node.js 22、pnpm 11 和冻结锁文件，执行类型检查、lint、单元/集成测试、生产构建及必要的端到端检查。

- **WHEN** 盘点测试，**THEN** 项目 **SHALL** 按单元、组件/集成、端到端和测试辅助设施划分职责，并根据实际边界重组目录、配置与命名，避免重复覆盖同一实现细节。

- **WHEN** 执行端到端测试，**THEN** 测试 **SHALL** 覆盖导航、响应式菜单、主题、语言对应路由、主要 CTA、申请异常态、动态详情、404 和关键 SEO 输出，并使用稳定选择器与隔离端口。
- **WHEN** 完成测试治理，**THEN** 必需测试 **SHALL** 在重复运行中无可复现 flaky；不得依赖增加重试来掩盖确定性问题。

## 4. 技术约束

### 4.1 技术栈

- **语言/框架**：Next.js 16 App Router、React 19、TypeScript 严格模式、Tailwind CSS 4。
- **运行环境**：Node.js 22、pnpm 11；统一使用 pnpm，不引入 npm/yarn 锁文件或命令。
- **渲染边界**：保持纯静态展示官网；默认 React Server Components，仅真实浏览器交互使用 Client Components。
- **测试框架**：沿用 Vitest、Testing Library 与 Playwright；除非设计阶段证明现有工具无法满足需求，不更换测试框架。
- **部署目标**：GitHub Pages 静态导出为正式发布路径；现有 Docker 构建能力在未明确废弃前必须保持可验证。

### 4.2 集成点

- **站点配置**：`src/config/site.ts` 是域名与公开申请入口的集中配置来源。
- **公开内容**：`src/content/` 与相关类型/校验是事实内容的代码内来源。
- **页面与组件**：`src/app/`、`src/components/pages/`、`src/components/site/`。
- **国际化与 SEO**：`src/i18n/`、metadata、robots、sitemap、JSON-LD 与动态路由静态参数。
- **质量与部署**：`package.json`、TypeScript/ESLint/Vitest/Playwright 配置、`.github/workflows/`、Dockerfile、`public/CNAME`。
- **外部服务**：飞书公开问卷仅作为外部跳转目标；官网不得代理提交或保存申请数据。

### 4.3 数据要求

- **存储**：不新增数据库、CMS、服务端持久化或用户账户系统。
- **个人信息**：官网不接收、不落库、不保存任何申请人个人信息。
- **公开事实**：只使用仓库现有且可公开的资料；授权不明的具名内容不得因文案优化而被推定为已授权。
- **双语规则**：中文为事实权威源；英文保持语义等价，并建立可维护的缺失/回退约束。
- **密钥**：不得提交 API key、token、口令或其他凭证；飞书公开问卷分享链接按项目约定不视为密钥。

### 4.4 非功能性需求

- **可访问性**：以 WCAG 2.2 AA 为最低目标，自动检测与人工键盘/屏幕阅读器语义检查结合。
- **性能**：代表性页面 Lighthouse Performance ≥ 90，其余质量类别 ≥ 95；图片、字体、脚本、CSS 和第三方资源应设置合理预算并避免明显回归。
- **兼容性**：Chromium、Firefox、WebKit 及代表性移动视口的核心旅程通过。
- **可靠性**：所有内部链接、静态路由和构建产物可验证；CI 必需检查失败时不得部署。
- **安全性**：最小权限工作流、无敏感信息泄漏、无已知高危/严重生产依赖漏洞；不作超出静态托管能力的虚假安全承诺。
- **可维护性**：权威配置、内容、设计 token、文档入口和测试职责保持单一且可发现。

### 4.5 审计与实施约束

- 必须先形成基线和问题清单，再按风险优先级实施；不得以大规模视觉重写掩盖未解决的内容、可访问性或发布问题。
- 可以调整信息架构、页面布局、组件、设计 token、文案、依赖、目录、测试和工作流，但每类重要变更需有回归证据。
- 现有用户改动与无关工作区内容必须保留；不得使用破坏性 Git 操作清理仓库。
- 域名变更联动规则严格遵循仓库 `AGENTS.md`：正式 canonical 一律使用带连字符的 `zgc-llm` 主域名。
- 既有 `docs/dev/` 规格是审计输入而非自动权威结论；设计阶段必须判断已完成、仍有效、冲突或过时状态。

## 5. 排除项

- **新增后台、数据库或 CMS**：违背纯静态官网边界，本轮不做。
- **官网内收集申请人信息**：继续由飞书公开问卷承担，官网不代理、不落库。
- **虚构或推定事实**：不创造未经资料支持的数据、荣誉、成员、合作、政策背书或公开授权。
- **无依据更换核心框架或部署平台**：保持 Next.js、React、pnpm 与 GitHub Pages；若审计发现必须迁移，需另行取得用户确认。
- **扩大到联盟内部业务系统**：本轮只治理公开官网及其代码、测试、文档与交付流水线。
- **替代法务、渗透测试或第三方认证**：可完成工程层面的隐私、安全和无障碍检查，但不宣称获得外部合规认证。
- **未经授权直接合并或生产部署**：交付目标是“发布就绪”；提交、推送、合并、DNS 变更和正式部署需用户另行明确授权。
- **重写 Git 历史或删除追溯资料**：历史文档优先标记状态或归档，除非确认无保留价值且删除范围明确。
