## 1. 概述

建设中关村自主大模型产业联盟官方网站一期，以联盟公信力和产业生态连接能力为核心，面向机构伙伴与专业用户展示联盟定位、重点工作、成员生态和阶段成果，并通过飞书表单承接合作意向。

网站以机构生态共建为首要转化目标，以专业用户加入为次要转化目标。网络安全生态作为联盟重点专项，建设独立专题页面，并在首页获得重点展示和入口。该专项保持厂商中立，不绑定单一基础模型，为不同基础模型厂商、技术路线和生态伙伴参与共建预留空间。

### 核心价值

- 对机构伙伴：快速理解联盟定位、合作方向、参与方式及合作价值，并提交生态共建申请。
- 对专业用户：了解开放计划、专项活动和参与方式，并进入专业用户申请流程。
- 对联盟运营方：建立统一、可信、可持续更新的公开信息窗口，为后续成员运营、项目协作和内容后台演进保留空间。

### 目标用户

1. 安全企业、基础模型厂商、高校、实验室、科研机构、行业组织、监管单位和重点行业客户。
2. 安全研究员、攻防团队、安全开发者、高校师生及其他专业研究人员。
3. 关注自主大模型产业、联盟动态、研究成果和生态合作的公众与媒体。

## 2. 需求与用户故事

### 2.1 全站信息架构

一期至少提供以下公开内容：

- 首页：联盟主张、核心价值、重点方向、重点专项、生态伙伴、最新动态和行动入口。
- 联盟介绍：联盟宗旨、定位、组织机制、发展方向及联系方式。
- 工作组与重点专项：展示联盟工作组或专项项目的定位、方向和参与方式。
- 网络安全生态专题：独立介绍战略背景、建设目标、生态闭环、参与角色、共建方向和加入方式。
- 生态伙伴或成员展示：按可公开范围展示合作单位、成员类型、标识和简介。
- 新闻、活动与成果：展示联盟动态、活动通知、行业观察、报告和典型成果。
- 加入联盟：说明机构参与条件、合作方式和权益，并引导至机构生态共建飞书表单。
- 专业用户加入：说明适用人群、参与方式和相关权益，并引导至专业用户飞书表单。
- 联系与合规信息：联系方式、隐私说明、必要的网站备案与版权信息。

具体导航命名可在视觉与内容设计阶段调整，但不得弱化两类申请入口和网络安全生态专题。

### 2.2 首页与转化入口

**用户故事：** 作为潜在机构伙伴，我希望在首次访问时快速理解联盟是谁、正在做什么、为什么值得合作，以及如何发起合作。

**验收标准：**

- 首页首屏明确展示联盟完整名称、核心主张和机构生态共建主按钮。
- 专业用户加入作为次级按钮或明确的次级入口呈现。
- 首页重点展示网络安全生态专项，并可进入独立专题页。
- 首页可展示重点方向、合作伙伴、新闻或成果摘要，并链接至对应详情或列表。
- 机构申请和专业用户申请入口在移动端同样清晰可用。

### 2.3 网络安全生态专题

**用户故事：** 作为网络安全领域的机构或专业人员，我希望理解该专项如何组织模型、专业用户、真实场景、深度数据、能力评测与产业落地，以及我可以通过什么方式参与。

**验收标准：**

- 专题页说明“模型发布 → 专业验证 → 场景落地 → 数据沉淀 → 模型增强 → 行业推广”的建设闭环。
- 介绍专业用户、机构伙伴、联盟、不同基础模型厂商、高校或实验室、运营方等参与角色。
- 专题定位和内容表达保持厂商中立，不将生态能力、参与资格或成果归属于某一个基础模型品牌。
- 展示可参与的共建方向，包括数据、任务、环境、专家、产品、评测或产业场景。
- 清楚说明数据授权、脱敏、高风险能力和攻防成果治理原则，不暗示强制交付原始数据。
- 同时提供机构生态共建和专业用户加入入口，机构入口视觉优先级更高。

### 2.4 飞书申请承接

**用户故事：** 作为有合作意向的访客，我希望从官网顺畅进入正式申请表单；作为运营人员，我希望无需修改页面代码即可替换表单目标地址。

**验收标准：**

- 机构生态共建与专业用户加入使用两个独立、可配置的飞书表单链接。
- 所有同类按钮使用一致的链接配置，避免页面之间出现不同目标地址。
- 外部跳转有明确文案和合理的浏览器行为，不伪装成站内账号或审核系统。
- 正式表单尚未提供时，网站使用明确的占位状态或统一联系渠道，不发布无效链接。
- 申请数据由飞书表单和飞书多维表格承接，官网一期不存储申请数据。

### 2.5 内容维护

**用户故事：** 作为开发维护人员，我希望通过项目内的 TypeScript 类型化配置更新新闻、活动、成果、成员和专项内容，并保持页面展示一致。

**验收标准：**

- 一期公开内容来源为项目内 TypeScript 类型化配置，不依赖 Payload CMS 内容集合或 Markdown 解析。
- 重复展示的站点名称、描述、联系方式、表单链接等信息集中配置。
- 新闻、活动、成果等条目具备统一的基本字段和排序方式。
- 内容缺失、列表为空或图片不可用时，页面仍有合理展示，不出现明显布局错误。
- 仓库现有 Payload CMS、PostgreSQL 和管理后台基础设施予以保留，不在一期拆除，也不作为一期内容发布的前置依赖。

### 2.6 响应式、可访问性与基础 SEO

**用户故事：** 作为使用手机或桌面设备访问的访客，我希望快速、清晰地浏览网站；作为通过搜索引擎进入的用户，我希望页面标题和摘要能够准确表达内容。

**验收标准：**

- 支持常见手机、平板和桌面视口，不产生阻断阅读的横向滚动。
- 主导航、移动导航、按钮、链接和表单跳转可通过键盘操作，并具有可辨识的焦点状态。
- 使用语义化标题结构，重要图片提供替代文本，正文对比度满足基本可读性要求。
- 全站具有站点级 metadata；主要页面具有独立标题、描述和规范 URL。
- 提供基础 sitemap 与 robots 配置；页面可被正常分享并显示合理的社交摘要。
- 页脚展示版权、联系方式、隐私说明入口，以及上线时要求的 ICP 备案信息。

- [ ] 首页以机构生态共建作为主行动入口，以专业用户加入作为次行动入口。

## 4. 技术约束

- 延续现有 Next.js 16 App Router、React 19、TypeScript 与 Tailwind CSS 4 技术栈，不在一期进行框架迁移。
- 保留 Payload CMS 3、PostgreSQL、现有管理后台路由和基础配置，为后续内容后台或业务系统演进预留能力。
- 一期页面内容使用项目内 TypeScript 类型化配置；不得为了展示内容而要求生产数据库必须可用。
- 飞书表单 URL 必须通过集中配置或环境变量管理，不将申请数据同步至官网数据库。
- 正式主域名沿用 `www.zgc-llm.org.cn`；`zgc-llm.cn` 和 `zgc-llm.net` 的跳转策略保持现有项目约定。
- 不在仓库中提交生产密钥、飞书访问凭证、数据库凭据或其他敏感信息。
- 内容发布前需确认联盟 Logo、视觉规范、正式合作单位名称及标识的公开授权。
- 上线前需补齐正式飞书表单、联系人、隐私说明、ICP备案信息及必要的合规文案。

## 5. 排除项

以下内容不属于官网一期范围：

- 用户注册、登录、会员中心和身份分层。
- 站内申请表单、申请材料上传、审核后台、申请进度和权益发放。
- 飞书开放平台 API 集成或官网与多维表格之间的自动数据同步。
- 面向运营人员的新闻、活动、成果或成员内容管理后台。
- 任务发布、数据上传、Agent 轨迹采集、专家复核和项目协作系统。
- 在线模型能力评测、Benchmark 执行、动态靶场和评测报告自动生成。
- 交易、付费、Token 或算力权益结算。
- 强制成员交付原始数据，或公开展示未经授权的攻防、漏洞及高风险内容。
- 中英双语、多站点或独立子域名体系，除非后续单独立项。
