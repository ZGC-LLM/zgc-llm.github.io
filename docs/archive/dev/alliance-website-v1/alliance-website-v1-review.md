## 3. 关键问题

### 已修正 P1-1：Group 1 并行依赖不安全

原计划允许内容配置、共享组件和 sitemap 同时修改，但后两者都依赖公开路由与申请目标接口，可能出现接口漂移和共享文件冲突。

**修正：** Group 1 拆成 1A（T-002 先冻结内容/配置接口）与 1B（T-003、T-004 并行）。

### 已修正 P1-2：组件测试与现有 Vitest 配置不兼容

现有 `vitest.unit.config.mts` 只包含 `tests/unit/**/*.test.ts` 且使用 Node 环境，无法直接运行 `.test.tsx` 与 Testing Library DOM 测试。

**修正：** T-003 明确扩展 include，并使用项目已安装的 `jsdom` 环境；design 同步记录。

### 已修正 P1-3：T-009 依赖与并行描述矛盾

T-009 依赖 T-008，却又声明可与 T-008 部分并行，容易造成跨页联调与页面实现冲突。

**修正：** T-009 改为所有公开页面完成后串行执行。

## 4. 重要遗漏与剩余风险

### P2-1：Payload 保留与“公开内容不依赖数据库”的边界需在基线中验证

公开页面设计不读取 Payload，但整个 Next.js 应用仍通过 `withPayload` 加载 Payload 配置，并要求合法 `PAYLOAD_SECRET`。生产应用保留管理后台时仍需要数据库；“不依赖数据库”应解释为内容发布与页面数据源不依赖 Payload 集合，而不是删除整个应用的数据库运行依赖。

**行动：** T-001 分别记录无数据库的单元/类型/lint 基线，以及带测试环境变量的 build/E2E 基线；若 build 会连接数据库，保留现有部署约束，不在本期做架构拆分。

### P2-2：正式内容和品牌资料是发布依赖

Logo、成员授权、新闻、联系人、备案信息和两份飞书表单尚未提供。技术页面可以完成，但不能以虚构数据代替。

**行动：** 按设计使用文字标识、空状态和不可用 CTA；T-010 将资料缺口列为发布阻塞项，而非代码缺陷。

### P2-3：外部表单的最终隐私责任仍需运营确认

官网可以说明跳转外部服务，但飞书表单字段、数据保留、负责人和表单内隐私告知不属于代码仓库能够验证的内容。

**行动：** 正式 URL 上线前由联盟运营方验收；自动化测试只校验 URL 协议和安全属性，不访问或提交真实表单。

## 5. 改进建议

- T-002 将公开路由定义为单一来源，导航、sitemap 和测试复用，避免路径漂移。
- `ExternalApplicationLink` 只接受解析成功的 HTTPS URL，并把缺失配置视为正常状态而非抛出运行时错误。
- 新闻正文保持受控内容块，不使用任意 HTML；未来接入 CMS 时另立数据迁移与富文本安全设计。
- 移动 E2E 在用例内显式设置 390 px 视口，当前不必扩大 Playwright project 矩阵。
- 视觉 Token 先使用临时、可替换的品牌体系；收到正式 VI 后只调整 Token 与素材，不重写页面结构。
